# ADR 2: Apply Worker-Level Partial Pipe-and-Filter

## Status

Accepted

## Context

TimeGrapher's real-time audio analysis uses a Pipe-and-Filter-style runtime flow at the worker level. In this view, worker/path elements are close to filters or final display consumers, and bounded connectors act as pipes.

![Worker-level partial Pipe-and-Filter](assets/worker-level-partial-pipe-and-filter.svg)

The inside of the Analysis worker is not split into separate pipes, queues, or threads. Its DSP/metrics path is a synchronous staged chain that processes each block on the same analysis thread.

Editable source: [worker-level-partial-pipe-and-filter.drawio](assets/worker-level-partial-pipe-and-filter.drawio)

This ADR records the decision to apply Pipe-and-Filter to the worker-level flow, but not to the worker-internal hot path.

## Decision

TimeGrapher applies Pipe-and-Filter as a worker-level partial application.

The worker-level flow is described by these filter/pipe boundaries:

- Filter: Input worker -> Pipe: bounded ring buffer -> Filter: Analysis worker
- Filter: Analysis worker -> Pipe: latest-wins frame scheduler -> Final display consumer: UI/render path
- Filter: Analysis worker -> Pipe: bounded recording queue -> Recording consumer: recording writer

Inside the Analysis worker, HPF, envelope, detector, and metrics/projectors are treated as an internal synchronous staged chain, not as worker-level filter/pipe boundaries.

## Rationale

Worker-level Pipe-and-Filter makes the responsibilities and concurrency boundaries for input, analysis, and rendering explicit. The ring buffer and latest-wins frame scheduler absorb producer/consumer speed differences and prevent UI rendering from directly blocking input capture or detection.

The inside of the Analysis worker is a hot path that must stay within the 125 ms beat budget at 28800 BPH. Splitting this internal chain into a full Pipe-and-Filter structure would add these costs:

- queueing, synchronization, and scheduling latency between stages
- buffer/message allocation or copying
- additional ordering logic for audio blocks and events
- higher state-management complexity for detector, sync PLL, and rolling metrics

Therefore, the design uses Pipe-and-Filter-style concurrency between workers while keeping the Analysis worker internals as a synchronous chain for real-time performance and structural simplicity.

## Consequences

Positive:

- The worker-level runtime flow is clearly explained from a Pipe-and-Filter viewpoint.
- Input, analysis, and UI rendering do not directly block one another.
- The Analysis worker hot path avoids unnecessary queue/thread overhead.
- Block ordering and detector/metrics state consistency remain simple.

Negative / trade-offs:

- Worker-internal stages do not provide parallel speedup.
- Analysis worker internal stages are not independent runtime components with their own pipe/connectors.
- This decision must be described as worker-level partial application, not full Pipe-and-Filter.
