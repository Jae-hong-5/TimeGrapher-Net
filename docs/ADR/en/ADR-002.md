# ADR 2: Apply Worker-Level Partial Pipe-and-Filter

TimeGrapher's real-time audio analysis runs as a worker-level flow in which worker/path elements act as filters or final display consumers and bounded connectors act as pipes. The following forces shaped how far Pipe-and-Filter should be applied.

- **Real-time budget**: At 28800 BPH each beat must be processed within a 125 ms budget, so the analysis hot path cannot absorb extra per-stage latency.
- **Concurrency boundaries**: Input capture, analysis, and UI rendering run at different speeds and must not block one another.
- **Recording**: Detected results must also reach a recording writer without stalling analysis.
- **Structural clarity**: The runtime flow should make input, analysis, and rendering responsibilities explicit for reviewers.

Inside the Analysis worker the DSP/metrics path is not split into separate pipes, queues, or threads; it is a synchronous staged chain that processes each block on the same analysis thread. This ADR records the decision to apply Pipe-and-Filter to the worker-level flow but not to the worker-internal hot path.

## Decision

We will apply Pipe-and-Filter as a worker-level partial application, with these filter/pipe boundaries:

- Filter: Input worker -> Pipe: bounded ring buffer -> Filter: Analysis worker
- Filter: Analysis worker -> Pipe: latest-wins frame scheduler -> Final display consumer: UI/render path
- Filter: Analysis worker -> Pipe: bounded recording queue -> Recording consumer: recording writer

We will keep the HPF, envelope, detector, and metrics/projectors inside the Analysis worker as an internal synchronous staged chain, not as worker-level filter/pipe boundaries.

## Rationale

Worker-level Pipe-and-Filter makes the responsibilities and concurrency boundaries for input, analysis, and rendering explicit. The ring buffer and latest-wins frame scheduler absorb producer/consumer speed differences and prevent UI rendering from directly blocking input capture or detection.

The inside of the Analysis worker is a hot path that must stay within the 125 ms beat budget at 28800 BPH. Keeping it as a synchronous chain avoids the latency, allocation, and state-management costs that a full Pipe-and-Filter split would add on the critical path.

Rejected alternatives:

- **Full Pipe-and-Filter inside the Analysis worker**: Rejected because splitting the internal chain into independent pipe/filter stages would add queueing, synchronization, and scheduling latency between stages, extra buffer/message allocation or copying, additional ordering logic for audio blocks and events, and higher state-management complexity for the detector, sync PLL, and rolling metrics — all on a path that must fit the 125 ms budget.
- **A single synchronous flow with no pipes between workers**: Rejected because input capture, analysis, and UI rendering would then block one another, and a slow UI frame could stall detection or input capture.

## Status

Accepted

## Consequences

Positive:

- The worker-level runtime flow is clearly explained from a Pipe-and-Filter viewpoint.
- Input, analysis, and UI rendering do not directly block one another.
- The Analysis worker hot path avoids unnecessary queue/thread overhead.
- Block ordering and detector/metrics state consistency remain simple.

Negative / trade-offs:

- Worker-internal stages do not provide parallel speedup.
- Analysis worker internal stages are not independent runtime components with their own pipe/connectors.
- This decision must be described as worker-level partial application, not full Pipe-and-Filter.

## Related Quality Attributes (QAS)

Quality attribute scenarios (QAS, Architectural Drivers document) this decision connects to:

- **QAS-2 Performance (Latency)**: The worker-level Pipe-and-Filter structure separates input, analysis, and rendering so they do not block one another, meeting the 125 ms beat budget at 28800 BPH.
- **QAS-4 Consistency**: The Analysis worker processes data in order on one analysis thread, keeping block ordering and detector/metrics state simple. The `AnalysisFrame` created in the same analysis cycle is delivered to the UI/render path, helping graphs and numeric displays in one frame come from the same source data.
