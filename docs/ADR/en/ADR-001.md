# ADR 1: Switch the UI Framework from Qt + C++ to Avalonia UI + .NET + C#

The existing application is built with Qt + C++. The following factors affected the decision on whether to keep that stack.

- **Personnel**: The current team has no members proficient in C++, while it does have members proficient in C#. Maintaining and extending the C++ code is effectively difficult.
- **Licensing**: Qt can be used for free under the LGPL, but distribution requires compliance obligations such as dynamic linking, license notices, and source availability.
- **Maintainability**: Manual memory management in C++ introduces defect risk, and the build system is complex because of CMake/MOC and platform-specific toolchains.
- **Cross-platform support**: Desktop and ARM64 embedded targets must be supported from a single codebase.

## Decision

We will switch the UI framework and application stack from **Qt + C++ to Avalonia UI + .NET + C#**.

## Rationale

- **Team capability**: Because the team has C# proficiency and no C++ proficiency, the C# stack is the most practical choice for maintainability and development speed.
- **Licensing**: Avalonia uses the MIT license, so it can be distributed freely without Qt's LGPL compliance obligations such as dynamic linking, notices, and source availability.
- **Productivity**: C# provides GC-based memory safety, a rich .NET standard library, and the NuGet ecosystem, which speeds up development.
- **Cross-platform consistency**: Avalonia's own renderer, Skia, guarantees pixel-level consistent rendering across desktop and ARM64 embedded environments.
- **UI customization**: The existing project's Qt Widgets approach, based on `.ui` forms and QSS styling, is mainly suited to visual styling such as colors, margins, and borders, making it difficult to change the control structure itself. By contrast, Avalonia has a lookless structure in which the entire appearance of each control can be redefined with `ControlTemplate`, and it provides CSS-like style selectors. This gives it greater freedom for UI customization.

Rejected alternatives:

- **Keep Qt + C++**: Rejected because the team lacks proficient C++ personnel, making maintenance difficult, and because the team would still have to carry LGPL compliance obligations.
- **.NET MAUI**: Rejected because Linux desktop and embedded support is weak.
- **Electron**: Rejected because including the full Chromium engine increases memory and disk usage, creating a burden for low-spec embedded devices.
- **Flutter**: Rejected because, although it has strong cross-platform support, its development language is Dart, which does not match the team's C# capability.

## Status

Accepted

## Consequences

Positive:

- Maintenance and development speed improve because the stack matches the team's existing capability in C#.
- The project avoids Qt's LGPL compliance obligations by using the MIT license.
- Desktop and embedded targets can be supported from a single codebase.

Negative / trade-offs:

- The initial development cost and schedule burden are high because of the full rewrite.
- There is a risk that behavior already validated in the existing C++ code may not be reproduced exactly in the new implementation.
- Including the .NET runtime may increase distribution size, startup time, and memory usage.
