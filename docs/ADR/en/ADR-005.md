# ADR 5: On-Device TinyML Signal-Quality Classifier Behind a Strategy Seam

The project plan's AI Feature asks for on-device intelligence that classifies signal quality (good / noisy / weak / unstable) in real time, without cloud services, on the Raspberry Pi. The system already exposes an injection seam for this ŌĆö `ISignalQualityClassifier` in `TimeGrapher.Core.Analysis.Quality` ŌĆö with a dependency-free `HeuristicSignalQualityClassifier` as the only implementation, not yet wired into the running app. We need to add a trained on-device model without breaking Core's zero-dependency rule (CI-enforced) and without letting a misclassification corrupt the measurements a timegrapher exists to report.

## Decision

- **We will implement the AI Feature's Signal Quality Classification as an on-device ONNX (TinyML) model behind the existing `ISignalQualityClassifier` Strategy seam**, not as new logic inside Core.
- **The ONNX implementation (`OnnxSignalQualityClassifier`) lives in a separate leaf project, `TimeGrapher.Inference`,** which references only `Core` and `Microsoft.ML.OnnxRuntime`. The trained model and a class-order sidecar are embedded in the assembly so the classifier is self-contained on Windows dev and the Raspberry Pi (`linux-arm64`).
- **The App selects the Strategy at the composition root** via `OnnxSignalQualityClassifier.LoadOrElse(LoadDefault, heuristic)`: the ONNX model when it loads, the heuristic when it cannot. The ONNX runtime is therefore referenced by `App`/`Verify` only, never by `Core`.
- **The classifier stays advisory and non-destructive**: its verdict annotates how far to trust the live readings and surfaces through the existing overlay; it never drops or alters a detected beat.
- **The model is trained offline by a dev-only tool (`tools/TimeGrapher.SignalQualityTrainer`) kept outside the solution**, so the ML.NET training stack never enters the shipped build. It distils the heuristic over a synthetic, self-labelled feature distribution.

## Rationale

- **Core must depend on nothing** (CI checks that `TimeGrapher.Core` has zero package and project references). Putting the ONNX runtime in a leaf project injected at the composition root keeps that guarantee while still shipping the model.
- **Swappability and testability**: the Strategy seam lets the heuristic, the ONNX model, or a future model be exchanged without touching the analysis pipeline, and lets the pipeline be tested without loading any model.
- **Advisory, not veto**: an earlier subsystem vetoed (dropped) "bad-data" beats; it was deliberately removed because dropping beats can hide the very irregularities a timegrapher must surface and can corrupt rate / beat error / amplitude. Keeping the classifier advisory means a misclassification can only mislabel trust, never the measurement.
- **On-device, offline**: the requirement forbids cloud dependence; `Microsoft.ML.OnnxRuntime` ships native ARM64 binaries, so the model runs on the Pi.
- **Rejected alternatives**: (a) placing the ONNX code in `Core` ŌĆö rejected, it breaks the zero-dependency rule and fails CI; (b) a cloud inference service ŌĆö rejected for the offline/latency/privacy requirement; (c) shipping only the heuristic ŌĆö rejected because it does not realize the on-device AI requirement; (d) reviving the drop/veto gate ŌĆö rejected because it destroys data (see above).

## Status

Accepted

## Consequences

Positive:

- Realizes the project plan's on-device AI Feature with a real trained model running on the Pi, while Core stays dependency-free and the choice of classifier stays swappable and retrainable.
- Because the classifier is advisory and non-destructive, a model error cannot move rate, beat error, or amplitude ŌĆö the measurement path is unaffected.
- A model that cannot be loaded degrades gracefully to the heuristic, so a corrupt or missing model never crashes startup.

Negative / trade-offs:

- Re-introduces the `Microsoft.ML.OnnxRuntime` dependency (and its native libraries) into the `App`/`Verify` build graph and the single-file publish size.
- The trained model is a checked-in binary artifact that must be regenerated by the dev-only trainer whenever the `SignalQualityFeatures` contract changes; the training stack is maintained outside the solution.
- The model is a learned approximation of the heuristic (Ōēł0.997 agreement), not an exact reproduction, so verdicts on borderline inputs can differ from the heuristic's hard thresholds.

## Related Quality Attributes (QAS)

- **QAS-5 Modifiability**: the Strategy seam confines the ONNX runtime to one leaf project injected at the composition root, so the model can be swapped or retrained with a small, localized change and Core stays portable.
- **QAS-1 Accuracy**: keeping the classifier advisory and non-destructive guarantees that detection and the computed metrics (rate, beat error, amplitude) are unchanged whether the model is on, off, or wrong.
- **QAS-3 Reliability**: the load-time fallback to the heuristic keeps the feature available even when the model is missing or unreadable.
