# ADR 3: Adopt the MVVM Pattern for the App's UI

ADR-001 chose Avalonia (.NET/C#) as the UI toolkit. We then had to decide how to organize the UI code. UI thread code easily piles up in one place, and when it does it becomes hard to change and hard to test.

We first intended to apply MVC, the pattern we had learned in class, but after review we chose MVVM. Both patterns split the screen (View) from the data and logic (Model); they differ only in the layer that connects the two. In MVC that layer is the Controller; in MVVM it is the ViewModel. The ViewModel holds the values and button actions to show on screen, and it is connected to the View automatically (this connection is called "data binding").

What shaped the decision:

- Avalonia is built around MVVM (data binding is built in). It has no separate Controller concept.
- UI logic must be automatically testable (ADR-004's TDD).
- Several people work at the same time, so UI design and UI logic must be separable without getting in each other's way.

## Decision

We will structure the App's UI with the **MVVM (Model-View-ViewModel)** pattern.

- View: the screen layout (XAML)
- ViewModel: the state and button actions shown on screen
- Model: the data and core logic (`TimeGrapher.Core`)

The View and ViewModel are connected by data binding, implemented directly with built-in .NET features and no extra library.

## Rationale

- **Fits the toolkit**: Avalonia is built for MVVM, so data binding alone connects the screen and the logic naturally. Forcing MVC on top works against this built-in support.
- **Easy to test**: the ViewModel is ordinary code separate from the screen, so it can be tested automatically without launching the UI.
- **Clear roles**: screen layout, UI logic, and data are separated, so changing one side rarely spills into the others.

Rejected alternatives:

- **MVC**: considered first because it was the most familiar, but Avalonia has no Controller concept, so it fits poorly.
- **No pattern, everything in the UI thread code**: the simplest option, but the logic is stuck to the screen so it can't be tested. The same value also ends up held separately in several UI elements, so one updates while another still shows the old value and they easily fall out of sync. It also causes frequent conflicts when several people work together.

## Status

Accepted

## Consequences

Positive:

- UI logic can be tested without launching the screen.
- Using Avalonia's data binding directly means less code that manipulates the screen by hand, and fewer bugs as a result.
- UI design and UI logic are easy to divide among several people.

Negative / trade-offs:

- If a data binding is wrong, the screen may just silently fail to update with no error, which can be hard to diagnose.
- We must keep an eye on the ViewModel so it doesn't accumulate too much logic.
- Moving the existing UI thread code over to MVVM took extra work.

## Related Quality Attributes (QAS)

Quality attribute scenarios (QAS, Architectural Drivers document) this decision connects to:

- **QAS-4 Consistency**: The ViewModel holds the screen state in one place and the View is bound to it, so values from the same source data are reflected consistently across displays (avoiding the drift that happens when several UI elements each hold the value separately).
- **QAS-5 Modifiability**: With View, ViewModel, and Model separated and the ViewModel independently testable, adding a new graph or display can keep the change surface small.
