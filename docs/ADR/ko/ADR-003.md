# ADR 3: App 화면 구조에 MVVM 패턴 채택

ADR-001에서 UI 도구로 Avalonia(.NET/C#)를 선택했다. 이제 화면 쪽 코드를 어떤 방식으로 정리할지 정해야 했다. UI 쓰레드 코드는 한곳에 뒤엉켜 커지기 쉽고, 그렇게 되면 고치기도 테스트하기도 어려워진다.

처음에는 수업에서 배워서 익숙한 MVC를 적용하고자 했으나, 검토 끝에 MVVM으로 정했다. 두 방식 모두 화면(View)과 데이터·로직(Model)을 나누는 점은 같고, 그 사이를 잇는 계층만 다르다. MVC는 Controller가, MVVM은 ViewModel이 그 역할을 맡는다. ViewModel은 화면에 보여줄 값과 버튼 동작을 담고, View와 자동으로 연결된다(이 연결을 "데이터 바인딩"이라 한다).

정하는 데 영향을 준 점:

- 우리가 쓰는 Avalonia는 MVVM에 맞춰 만들어졌다(데이터 바인딩을 기본 제공). 따로 Controller 개념은 없다.
- 화면 로직을 자동으로 테스트할 수 있어야 한다(ADR-004의 TDD).
- 여러 명이 동시에 작업하므로, 화면 디자인과 화면 로직이 서로 방해 없이 나뉘어야 한다.

## Decision

우리는 App의 화면을 **MVVM(Model-View-ViewModel)** 방식으로 구성할 것이다.

- View: 화면 모양(XAML)
- ViewModel: 화면에 보여줄 상태와 버튼 동작
- Model: 데이터와 핵심 로직(`TimeGrapher.Core`)

View와 ViewModel은 데이터 바인딩으로 연결하고, 별도 라이브러리 없이 .NET 기본 기능으로 직접 구현한다.

## Rationale

- **도구와 잘 맞는다**: Avalonia가 MVVM을 전제로 만들어져, 데이터 바인딩만으로 화면과 로직을 자연스럽게 잇는다. MVC를 억지로 얹으면 이 기본 기능과 어긋난다.
- **테스트하기 쉽다**: ViewModel은 화면과 분리된 일반 코드라, 화면을 띄우지 않고도 자동으로 테스트할 수 있다.
- **역할이 나뉜다**: 화면 모양·화면 로직·데이터가 분리되어, 한쪽을 고쳐도 다른 쪽으로 잘 번지지 않는다.

기각된 대안

- **MVC**: 가장 익숙해 처음 고려했지만, Avalonia에는 Controller라는 개념이 없어 잘 맞지 않는다. MVC는 원래 웹(요청-응답) 쪽에 더 어울린다.
- **패턴 없이 UI 쓰레드 코드에 다 넣기**: 가장 간단하지만, 로직이 화면에 붙어 있어 테스트가 불가능하다. 또 같은 값을 여러 화면 요소가 제각기 들고 있게 되어 한쪽만 갱신되고 다른 쪽은 옛 값을 보여주는 식으로 일관성이 깨지기 쉽다. 여러 명이 같이 작업할 때 충돌도 잦다.

## Status

승인됨(Accepted)

## Consequences

긍정적:

- 화면을 띄우지 않고도 화면 로직을 테스트할 수 있다.
- Avalonia의 데이터 바인딩을 그대로 써서 화면을 직접 조작하는 코드가 줄고, 그만큼 버그도 줄어든다.
- 화면 디자인과 화면 로직을 여러 명이 나눠 작업하기 쉽다.

부정적 / 트레이드오프:

- 연결을 자동으로 해주는 라이브러리를 쓰지 않으므로, 기본 연결 코드를 직접 작성해야 한다.
- 데이터 바인딩이 잘못되면 오류 없이 조용히 화면만 안 바뀔 수 있어 원인을 찾기 어려울 때가 있다.
- ViewModel에 로직이 과하게 쌓이지 않도록 계속 신경 써야 한다.
- 기존 UI 쓰레드 코드를 MVVM 방식으로 옮기는 작업이 필요했다.
