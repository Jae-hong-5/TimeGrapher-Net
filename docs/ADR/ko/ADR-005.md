# ADR 5: Strategy Seam 뒤의 온디바이스 TinyML 신호품질 분류기

프로젝트 계획서의 AI Feature는 클라우드 없이 라즈베리파이에서 신호 품질(good / noisy / weak / unstable)을 실시간 분류하는 온디바이스 지능을 요구한다. 시스템에는 이를 위한 주입 seam이 이미 있다 — `TimeGrapher.Core.Analysis.Quality`의 `ISignalQualityClassifier` — 그리고 무의존 구현인 `HeuristicSignalQualityClassifier`가 유일한 구현으로 존재하나 아직 실행 앱에 배선되지 않았다. 우리는 Core의 무의존 규칙(CI 강제)을 깨지 않고, 오분류가 타임그래퍼가 보고해야 할 측정값을 오염시키지 않게 하면서 학습된 온디바이스 모델을 추가해야 한다.

## Decision

- **AI Feature의 Signal Quality Classification을 Core 내부의 새 로직이 아니라, 기존 `ISignalQualityClassifier` Strategy seam 뒤의 온디바이스 ONNX(TinyML) 모델로 구현한다.**
- **ONNX 구현(`OnnxSignalQualityClassifier`)은 별도 리프 프로젝트 `TimeGrapher.Inference`에 둔다.** 이 프로젝트는 `Core`와 `Microsoft.ML.OnnxRuntime`만 참조한다. 학습된 모델과 클래스 순서 사이드카를 어셈블리에 임베드해, Windows 개발 환경과 라즈베리파이(`linux-arm64`)에서 분류기가 자기 완결적이 되게 한다.
- **App이 합성 루트에서 Strategy를 선택한다** — `OnnxSignalQualityClassifier.LoadOrElse(LoadDefault, heuristic)`: 모델이 로드되면 ONNX, 안 되면 휴리스틱. 따라서 ONNX 런타임은 `App`/`Verify`만 참조하고 `Core`는 절대 참조하지 않는다.
- **분류기는 advisory(권고용)이며 비파괴적이다**: 판정은 라이브 읽기를 얼마나 신뢰할지를 주석으로 표시해 기존 오버레이로 노출될 뿐, 검출된 beat를 버리거나 바꾸지 않는다.
- **모델은 솔루션 밖의 dev 전용 도구(`tools/TimeGrapher.SignalQualityTrainer`)로 오프라인 학습한다.** 그래서 ML.NET 학습 스택은 배포 빌드에 들어가지 않는다. 이 도구는 합성·자기 라벨링한 피처 분포에서 휴리스틱을 distill한다.

## Rationale

- **Core는 아무것도 의존하면 안 된다**(`TimeGrapher.Core`가 패키지·프로젝트 참조 0임을 CI가 검사). ONNX 런타임을 합성 루트에서 주입하는 리프 프로젝트에 두면 그 보장을 유지하면서도 모델을 함께 배포할 수 있다.
- **교체 가능성과 테스트 용이성**: Strategy seam 덕분에 휴리스틱·ONNX 모델·미래 모델을 분석 파이프라인을 건드리지 않고 교체할 수 있고, 모델 로드 없이 파이프라인을 테스트할 수 있다.
- **veto가 아니라 advisory**: 이전에 "나쁜 데이터" beat를 veto(폐기)하던 하위 시스템이 있었으나 의도적으로 제거했다 — beat를 버리면 타임그래퍼가 드러내야 할 이상 자체를 숨길 수 있고 rate·beat error·amplitude를 오염시킬 수 있기 때문이다. 분류기를 advisory로 유지하면 오분류가 측정값이 아니라 신뢰도 표시만 잘못 줄 수 있다.
- **온디바이스·오프라인**: 요구사항이 클라우드 의존을 금한다. `Microsoft.ML.OnnxRuntime`은 네이티브 ARM64 바이너리를 제공하므로 모델이 Pi에서 구동된다.
- **기각된 대안**: (a) ONNX 코드를 `Core`에 두기 — 무의존 규칙을 깨고 CI에 실패하므로 기각; (b) 클라우드 추론 서비스 — 오프라인/지연/프라이버시 요구로 기각; (c) 휴리스틱만 배포 — 온디바이스 AI 요구를 실현하지 못해 기각; (d) drop/veto 게이트 부활 — 데이터를 파괴하므로 기각(위 참조).

## Status

Accepted

## Consequences

긍정적:

- 학습된 실제 모델을 Pi에서 구동해 프로젝트 계획서의 온디바이스 AI Feature를 실현하면서도, Core는 무의존을 유지하고 분류기 선택은 교체·재학습 가능하다.
- 분류기가 advisory·비파괴적이라 모델 오류가 rate·beat error·amplitude를 움직일 수 없다 — 측정 경로는 영향받지 않는다.
- 로드할 수 없는 모델은 휴리스틱으로 우아하게 폴백하므로, 손상·누락 모델이 startup을 죽이지 않는다.

부정적 / 트레이드오프:

- `Microsoft.ML.OnnxRuntime` 의존성(그리고 네이티브 라이브러리)을 `App`/`Verify` 빌드 그래프와 단일 파일 publish 크기에 다시 들인다.
- 학습된 모델은 체크인된 바이너리 산출물이며, `SignalQualityFeatures` 계약이 바뀌면 dev 전용 트레이너로 재생성해야 한다. 학습 스택은 솔루션 밖에서 관리한다.
- 모델은 휴리스틱의 학습된 근사(≈0.997 일치)이지 정확한 재현이 아니므로, 경계 입력에서는 휴리스틱의 하드 임계값과 판정이 다를 수 있다.

## Related Quality Attributes (QAS)

- **QAS-5 Modifiability**: Strategy seam이 ONNX 런타임을 합성 루트에서 주입하는 리프 프로젝트 하나에 가두므로, 작고 국소적인 변경으로 모델을 교체·재학습할 수 있고 Core는 이식성을 유지한다.
- **QAS-1 Accuracy**: 분류기를 advisory·비파괴적으로 유지하면 모델이 켜지든 꺼지든 틀리든 검출과 계산 지표(rate·beat error·amplitude)가 변하지 않음이 보장된다.
- **QAS-3 Reliability**: 로드 시점의 휴리스틱 폴백이 모델 누락·판독 불가 시에도 기능을 계속 제공한다.
