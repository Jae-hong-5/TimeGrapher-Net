# ADR 1: UI 프레임워크를 Qt + C++ 에서 Avalonia UI + .NET + C# 로 전환

기존 애플리케이션은 Qt + C++ 로 구축되어 있다. 다음 요인들이 스택 유지 여부에 대한 결정에 영향을 미쳤다.

- **인력**: 현재 팀 내에 C++ 숙련자가 없고, C# 숙련자는 보유하고 있다. C++ 코드의 유지보수·확장이 사실상 어렵다.
- **라이선스**: Qt를 무료(LGPL)로 쓸 수 있으나, 배포 시 동적 링크·라이선스 고지·소스 제공 등 LGPL 준수 의무가 따른다.
- **유지보수**: C++ 수동 메모리 관리로 인한 결함 위험이 있고, 빌드 체계(CMake/MOC, 플랫폼별 툴체인)가 복잡하다.
- **크로스 플랫폼**: 데스크톱과 ARM64 임베디드를 단일 코드베이스로 지원해야 한다.

## Decision

우리는 UI 프레임워크와 애플리케이션 스택을 **Qt + C++ 에서 Avalonia UI + .NET + C# 로 전환할 것이다.**

## Rationale

- **팀 역량**: 팀에 C# 숙련자가 있고 C++ 숙련자가 없으므로, C# 스택이 유지보수·개발 속도 면에서 가장 현실적이다.
- **라이선스**: Avalonia는 MIT 라이선스라 Qt의 LGPL 준수 의무(동적 링크·고지·소스 제공) 없이 자유롭게 배포할 수 있다.
- **생산성**: C#의 GC 기반 메모리 안전성, 풍부한 .NET 표준 라이브러리와 NuGet 생태계로 개발이 빠르다.
- **크로스 플랫폼 일관성**: Avalonia 자체 렌더러(Skia)로 데스크톱·ARM64 임베디드에서 픽셀 단위로 동일한 렌더링을 보장한다.
- **UI 커스터마이징**: 기존 프로젝트가 사용 중인 Qt Widgets(`.ui` 폼 + QSS 스타일링)는 색·여백·테두리 등 외형 꾸미기 위주여서 컨트롤 구조 자체를 바꾸기 어렵다. 반면 Avalonia는 모든 컨트롤이 `ControlTemplate`으로 외형을 통째로 재정의할 수 있는(lookless) 구조이고 CSS 유사 스타일 셀렉터를 제공하므로, UI 커스터마이징 자유도가 더 높다.

기각된 대안

- **Qt + C++ 유지**: 팀에 숙련 인력이 없어 유지보수가 어렵고 LGPL 준수 의무도 계속 떠안아야 하므로 기각.
- **.NET MAUI**: Linux 데스크톱/임베디드 지원이 약해 기각.
- **Electron**: 크롬 엔진을 통째로 포함해 메모리·디스크 사용량이 커서 사양이 낮은 임베디드 기기에 부담이 크므로 기각.
- **Flutter**: 크로스 플랫폼 강점은 있으나 개발 언어가 Dart라 팀 역량(C#)과 맞지 않아 기각.

## Status

승인됨(Accepted)

## Consequences

긍정적:

- 팀 보유 역량(C#)에 맞아 유지보수·개발 속도가 개선된다.
- Qt의 LGPL 준수 의무에서 벗어난다(MIT 라이선스).
- 단일 코드베이스로 데스크톱·임베디드를 함께 지원한다.

부정적 / 트레이드오프:

- 전면 재작성에 따른 초기 개발 비용과 일정 부담이 크다.
- 기존 C++ 코드에서 검증된 동작이 새 구현에 정확히 재현되지 않을 위험이 있다.
- .NET 런타임 포함으로 배포 크기·시작 시간·메모리 사용량이 증가할 수 있다.

## Related Quality Attributes (QAS)

이 결정과 연결되는 품질 속성 시나리오(QAS, Architectural Drivers 문서):

- **QAS-5 Modifiability**: 팀이 보유한 C# 역량과 .NET/NuGet 생태계로 유지보수·기능 추가 속도를 확보해, 정해진 일정 안에서 기능을 늘리는 확장성을 뒷받침한다.
- **QAS-6 Usability**: Avalonia의 자체 렌더러(Skia)가 Pi 5의 1280×800 터치스크린에 GUI를 일관되게 그리고, `ControlTemplate` 기반 커스터마이징으로 작은 화면에 맞춘 UI 구성을 가능하게 한다.
- **QAS-2 Performance(트레이드오프)**: .NET 런타임을 포함하면 메모리·시작 시간이 늘 수 있어, Pi 5의 지연 예산을 지키는지 함께 확인해야 한다.
