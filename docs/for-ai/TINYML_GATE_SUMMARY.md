# TinyML 적용 방향

## 결정 요약

TinyML을 검출기 전체에 바로 적용하지 않기로 한 이유는 위험이 너무 크기 때문이다.

현재 검출기는 단순히 "소리가 있다/없다"를 보는 역할이 아니라, BPH 탐지와 PLL 동기화까지 함께 맡고 있다. 이 부분을 통째로 ML로 바꾸면 실시간성, 검증 범위, 실패했을 때의 영향이 모두 커진다. 특히 모델이 잘못 판단하면 화면 숫자만 틀리는 수준이 아니라 동기화 자체가 흔들릴 수 있다.

그래서 선택한 방향은 **기존 검출기는 그대로 두고, 검출기가 찾은 후보 이벤트 중 잡음으로 보이는 것만 마지막에 걸러내는 구조**다.

```text
검출기 -> 후보 이벤트 -> IBeatEventGate -> 계산/화면 표시
```

즉 TinyML은 "검출기 대체"가 아니라 **후보 이벤트 필터**로 붙인다. 기존 알고리즘이 하던 BPH 탐지와 PLL 동기화는 그대로 유지하고, TinyML은 계산과 화면 표시로 넘어가기 직전에 후보를 통과시킬지 버릴지만 판단한다.

## 해결하려던 문제

시계 소리에는 진짜 틱/톡 말고도 짧고 강한 잡음이 섞일 수 있다. 이런 잡음이 A 이벤트처럼 잡히면 rate, beat error, amplitude가 순간적으로 튈 수 있다.

문제의 핵심은 검출기 전체가 틀렸다는 것이 아니라, **검출기가 만든 후보 중 일부 잡음 후보가 계산 결과를 오염시킨다는 점**이다.

따라서 필요한 것은 전체 검출 알고리즘 교체가 아니라, 후보 이벤트가 계산으로 넘어가기 전에 한 번 더 확인하는 안전한 필터다.

## 검토한 선택지

1. 검출기 전체를 ML로 교체
   - 피했다.
   - BPH 탐지, PLL 동기화, A/C 이벤트 타이밍까지 모두 영향받는다.
   - 검증해야 할 범위가 너무 넓고, Raspberry Pi 실시간 성능 위험도 커진다.

2. ML을 `Core` 안에 직접 넣기
   - 피했다.
   - ONNX Runtime 같은 무거운 의존성이 `Core`에 들어오면 현재의 독립적인 분석 구조가 깨진다.
   - Windows/Linux/Raspberry Pi 배포 부담도 커진다.

3. 화면에서만 잡음 이벤트 숨기기
   - 피했다.
   - 화면만 숨기면 내부 계산, 로그, Verify 결과는 여전히 오염될 수 있다.
   - 계산과 화면 표시로 넘어가기 전 지점에서 걸러야 한다.

4. 후보 이벤트 게이트 추가
   - 선택했다.
   - 기존 검출과 동기화는 보존하고, 후보 이벤트만 통과/차단한다.
   - TinyML 모델이 틀려도 새 이벤트를 만들거나 이벤트 시간을 바꾸지 못하므로 실패 영향이 제한된다.

## 현재 만들어둔 구조

핵심은 `IBeatEventGate`다.

`IBeatEventGate`는 후보 이벤트를 받아서 통과시킬지 차단할지만 결정한다. 일부러 다음 기능은 못 하게 막아두었다.

- 새 이벤트 생성 불가
- 이벤트 시간 수정 불가
- BPH/PLL 동기화 입력 변경 불가
- 검출기 자체 대체 불가

이 제한이 중요하다. TinyML을 붙이더라도 모델은 계산에 들어갈 후보를 줄일 수 있을 뿐, 검출기의 시간 기준이나 동기화 상태를 직접 흔들 수 없다.

현재 같은 위치에 붙어 있는 구현은 `PllMatchGate`다. UI의 `PLL Event Veto (impulse rejection)` 옵션을 켜면, PLL 기준으로 위상이 맞지 않는 이벤트를 계산과 화면 표시 전에 버린다.

아직 실제 TinyML/ONNX 모델은 붙어 있지 않다. 지금 상태는 TinyML 완성 상태가 아니라, **나중에 TinyML을 안전하게 꽂을 수 있는 자리를 먼저 만들어둔 상태**다.

## TinyML을 붙이면 기대하는 효과

TinyML 모델은 후보 이벤트 주변 파형을 보고 이 후보가 진짜 틱/톡인지, 잡음인지 판단한다.

기대 효과는 다음과 같다.

- 임펄스 잡음 때문에 rate/beat error/amplitude가 튀는 문제 줄이기
- 잘못 잡힌 A 이벤트와 그에 딸린 C 이벤트가 계산을 오염시키는 문제 줄이기
- 약한 신호나 노이즈 많은 환경에서 진짜 이벤트만 남길 가능성 높이기
- `Verify`에서 `off`, `pll`, `onnx` 방식을 비교해 효과 측정하기

## 학습과 실행 방식

TinyML을 실제로 쓰려면 학습이 필요하다. 다만 앱이나 Raspberry Pi에서 실시간으로 학습하는 구조는 아니다.

역할은 이렇게 나눈다.

- PC: 데이터 준비, 모델 학습, `.onnx` 파일 생성
- 앱/Raspberry Pi: 학습된 `.onnx` 모델을 로드해서 후보 이벤트마다 추론

입력 데이터는 전체 오디오가 아니라, 검출기가 찾은 후보 이벤트 주변 파형이다. `BeatWindowFeatures`는 이 주변 파형을 128개 숫자로 바꾸는 특징 추출기다. 이 계약을 공통으로 두면 Python 학습 쪽과 C# 추론 쪽이 같은 입력 형식을 쓸 수 있다.

## Raspberry Pi에서 쓰는 방식

Raspberry Pi에서도 쓸 수 있다. 단, Pi에서는 학습이 아니라 추론만 한다.

전제는 다음과 같다.

- 64-bit ARM Linux 환경이어야 한다.
- 배포는 `linux-arm64` self-contained publish가 안전하다.
- ONNX Runtime은 CPU 추론 기준으로 생각한다.
- 모델은 작게 유지해야 한다.
- 이 기능은 모든 오디오 샘플마다 도는 것이 아니라, 검출기가 찾은 후보 이벤트마다 한 번씩 돈다.
- 실제 사용 가능 여부는 Pi에서 `Verify` 비교와 지연 시간 측정으로 확인해야 한다.

즉 결론은 **학습은 PC에서 하고, Pi에는 학습된 모델과 추론 코드만 넣어 쓰는 방식**이다.

## 나중에 붙일 프로젝트 구조

미래의 ONNX/TinyML 구현은 `TimeGrapher.Inference` 같은 별도 추론 프로젝트로 들어갈 가능성이 높다.

```text
App -> Inference -> Core
Verify -> Inference -> Core
Core -> ML 런타임에 직접 의존하지 않음
```

이렇게 하면 ONNX Runtime 같은 무거운 의존성은 추론 프로젝트에만 두고, `Core`는 지금처럼 독립적으로 유지할 수 있다.

## 핵심 파일

| 파일 | 의미 |
| --- | --- |
| `src/TimeGrapher.Core/Detection/Scoring/IBeatEventGate.cs` | 후보 이벤트 통과/차단 인터페이스 |
| `src/TimeGrapher.Core/Analysis/BeatEventGateHost.cs` | 검출기와 계산/표시 사이에서 게이트 실행 |
| `src/TimeGrapher.Core/Detection/Scoring/BeatWindowFeatures.cs` | 이벤트 주변 파형을 128개 숫자로 바꾸는 특징 추출기 |
| `src/TimeGrapher.Core/Detection/Scoring/PllMatchGate.cs` | 현재 사용 중인 기본 게이트 |
| `src/TimeGrapher.Verify/AdverseScenarios.cs` | `off`/`pll` 비교와 미래 `onnx` 예약 |

## 용어 정리

| 피할 표현 | 더 쉬운 표현 |

## 결론

TinyML 적용의 핵심은 **검출은 기존 알고리즘이 계속 맡고, TinyML은 잡음 후보를 마지막에 걸러서 측정값 안정성을 높이는 것**이다.

이 방식은 ML을 적용할 여지를 만들면서도, 검출기 전체를 바꾸는 위험과 `Core`에 ML 런타임을 직접 넣는 부담을 피한다.
